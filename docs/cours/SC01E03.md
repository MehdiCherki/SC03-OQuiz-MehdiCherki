## Merise

Méthodologie complète de gestion de gestion de projet.

On va ici se servir de la partie "conception de base de données" de cette méthodologie.

Méthodologie française, assez académique, ce qui se répercute sur les règles imposés par certains diagrammes.

Vocabulaire :
- **SGBD** = Système de gestion de baseS de donnéeS
- **BDD** = Base de données
- **SGBDR** = SGBD + Relationnel (= tables)


MERISE =
- **MCD** : Modèle conceptuel de données
  - objectif : réfléchir à "que dois-je stocker" ?
  - exercice le plus normalisé : règles et un vocabulaire précis
  - modèle pour représenter les données de la réalité : quelles sont les données du problèmes 
  - cible : investisseur, pour mamie
- **MLD** : Modèle logique de données
  - objectif : réfléchir à "comment je vais le stocker" ?
  - on trouve souvent plusieurs formats pour représenter nos données
  - cible : les techos, les devs
- **MPD** : Modèle physique de données
  - objectif : réfléchir à "stockons les données"
  - choix du SGBD et implémentation
  - cible : le SGBD

Deux approches complémentaires : 
- **Recueil de données**
  - faire l'inventaire de toutes les données à stocker
  - brouillon de préparation / déstructuré
- **Dictionnaire de données**
  - généralement sous forme de tableau ou l'on présente l'intégralité des champs et des types de chaque table


## Postgres en pratique

- Se connecte à Postgres en tant que super utilisateur
  - `sudo -i -u postgres psql`

- Créer un utilisateur
  - `CREATE ROLE oquiz WITH LOGIN CREATEDB PASSWORD 'oquiz';`
  - (on aura besoin du droit `CREATEDB` pour Prisma plus tard)

- Créer une base de données
  - `CREATE DATABASE oquiz WITH OWNER oquiz;`

- Se déplacer sur la base
  - `\c oquiz oquiz`

- Exécuter le script 
  - `\i <chemin_du_script>` (penser à drag&drop)

- Vérifier
  - `\dt`
  - `\d user`

## API REST 

Une API = Application Programming Interface

API Web = via HTTP

API REST = une API Web qui respecte un certains nombres de conventions : 
- accepte et renvoie du JSON
- conventions sur les verbes HTTP
  - `GET` = obtenir de l'information
  - `POST` = créer de la données sur le serveur
  - `PATCH` = modifier (partiallement) une ressource
  - `PUT` = remplacement (complet) d'une ressource
  - `DELETE` = suppression
- conventions sur les chemins
  - nom des ressources au pluriel (ex : `GET /api/artists/:id`)
  - notion de hierarchie sur les ressources récupérer (ex : `/playlists/{playlist_id}/tracks` pour récupérer les tracks de tel playlist)

Exemple d'API très bien documentée : [Spotify](https://developer.spotify.com/documentation/web-api)
