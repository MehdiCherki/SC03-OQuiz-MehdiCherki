## Exécution du TypeScript 

Avec Node :

Option 1 : 
- compiler le code TS vers JS à l'aide du transpilateur `TSC`
- plutôt cette approche là en production (on le fera cette aprem' avec l'API)

Option 2 : 
- Utiliser de la compilation à la volée avec `TSX`
- avantages : pratique et mature
- inconvénient : pas prévu pour la production (bootstrap long et transpilation à la volée)

Option 3 : 
- Node 22.18 (aout 2025 !) : pas besoin de compiler avec TSC, pas besoin de TSX, on a tout à la volée
- avantage : pratique pour la démo de cours 
- inconvénient : pas encore mature, on préfère toujours compiler pour la mise en production

## Types de base

Types primitifs
- `string`
- `number`
- `boolean`

Types nullable
- `undefined` : type undefined
- `null` : type null
- `void` : "cette fonction ne retourne rien"

Autres
- `any` : n'importe quel type --> on peut tout faire avec la varialbe, comme en JS
- `unknown` : "je ne maitrise pas ce qu'il y a dans la variable" --> vérification nécessaire des types à faire avant de s'en servir
- `never` : "il n'y a RIEN dans la variable"


## Fonction `.map()`

Entrée : tableau d'ELEMENTS
Sortie : tableau d'ELEMENTS (a qui on a fait subir une opération)

```js
const fruits = ["kiwi", "mango", "apple"];

const fruitsUpperCased = fruits.map(fruit => fruit.toUpperCase());

console.log(fruits); // ["kiwi", "mango", "apple"]
console.log(fruitsUpperCased); // [ 'KIWI', 'MANGO', 'APPLE' ]
```

## Commandes NPM

- `npm init` : créer le package.json
- `npm install xxxxx` : permet d'installer un package =
  - le déclarer dans le `package.json`
  - le télécharger dans les `node_modules`
- `npm install` : permet d'installer tous les packages déclarés dans le `package.json`
- `npm install --save-dev xxxxx` : déclarer comme dépendance de dev 
  - pour le local
  - pour produire le code qui sera ensuite exécuter en production
- `npx <command>` : exécuter une commande dont l'exécutable (binaire) se trouve dans les `node_modules`

Pourquoi pas installer TS au global (voire même avec `--global`) : 
- car `npm create vite@latest` gère sa propre config TS

Notre `package.json` global nous servira uniquement à : 
- déclarer les métadonnées du projet
- utiliser l'option `--prefix api` ou `--prefix client` pour gérer les différents services

## Prisma

```json
// A partir du 'schema.prisma', génère un/des fichier(s) de migrations (SQL) et les applique sur la BDD
"db:migrate:dev": "prisma migrate dev",

// Applique les migrations non encore appliquées sur la BDD
"db:migrate:deploy": "prisma migrate deploy",

// Supprime le schéma de la base de données, réinitialise et réapplique toutes les migrations
"db:migrate:reset": "prisma migrate reset --force",

// A partir du 'schema.prisma', génère le client Prisma
"db:generate": "prisma generate",

// Exécute le script de seed pour insérer des données initiales dans la base de données
"db:seed": "tsx --env-file=.env ./src/models/seeding.ts",

// Réinitialise la base de données (migrations + seed)
"db:reset": "npm run db:migrate:reset && npm run db:seed",

// Ouvre Prisma Studio (interface graphique pour visualiser/modifier la BDD)
"db:studio": "prisma studio",
```
