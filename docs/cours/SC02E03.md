## Nomenclature de tests

Les tests automatisés regroupent plusieurs types de tests : 
- **Tests unitaires** = on test une fonction isolée
  - ex : `function isValidPassword(password)`
- **Tests d'intégration** = on test une sous partie d'un système
  - ou component testing 
  - ou specification testing
  - ex : tester la route `GET /api/users`
- **Tests e2e** = on test l'intégralité d'une fonctionnalité, du point de vu de l'utilisation
  - ou test end-to-end
  - ou test bout-en-bout
  - ex : tester le fonctionnement de la page `/login`

Vocabulaire : 
- Contract Tests = vérifier que le code respecte le contrat
- Snapshot Testing = vérifier que le rendu actuel est le même que précédemment
- Non-regression Tests = éviter les régressions
- UI Testing = tester des composant UI, à l'aide d'un [storybook](https://storybook.js.org/) par exemple

Beaucoup d'abus de language sur les tests, mais l'idée reste la même : 
- éviter les régressions
- documenter le système
- s'assurer de la qualité logiciel
- faciliter l'intégration continue


## Jeu d'essai de tests

Les tests automatés sont un bon complément à un **jeu d'essai de tests** manuel.

Jeu d'essai de tests = Checklist de fonctionnalités à vérifier avant de re-déployer le système

On appelle aussi ce tableau un **plan de test**

### Exemple

| Cas                             | Données d’entrée                                      | Attendu                                                 |
| ------------------------------- | ----------------------------------------------------- | ------------------------------------------------------- |
| Login avec credentials valides  | email: `user@test.com` <br> password: `password123`   | ✅ Redirection tableau de bord                           |
| Login avec mauvais mot de passe | email: `user@test.com` <br> password: `wrongpass`     | ❌ Message "mot de passe invalide"                       |
| Inscription nouvelle adresse    | email: `newuser@test.com` <br> password: `newpass123` | ✅ Création compte + redirection                         |
| Token expiré                    | token expiré en DB                                    | ❌ Redirection vers login avec message "Session expirée" |


### Petit point TP

Le jury appréciera toujours que vous testiez vos applications, et surtout que vous ayez prévu cette partie testing dans votre workflow de développement.

Pour faire simple sur ce qu'on attend au TP sur le jeu d'essai :
- en DWWM = expliquer comment vous vérifiez le fonctionnement d'une fonctionnalité, en partant d'une situation donnée, en prévoyant le résultat attendu et en le comparant au résultat réellement obtenu
- en CDA = expliquer la même chose, en expliquant en plus à quel moment du processus de développement vous devez faire ces vérifications et pourquoi, ainsi que comment vous pouvez les automatiser (tests unitaires, d'intégration, end-to-end, etc.)
- 
Par exemple, à défaut d'écrire des tests E2E (long, setup complexe), proposer un jeu d'essai de test précis. Et expliciter à quel moment il doit être joué 
(ex : juste avant le déploiement à la fin d'un sprint)

## Mise en place de tests

On a besoin de deux choses : 
- **test runner** : l'exécutable qui lance les tests 
  - ex : `node:test` (natif à node mais récent) (`node --test`)
  - ex : `mocha`
  - ex : `jest`
  - ex : `vitest`
- **bibliothèque d'assertion** : librairie qui permet de vérifier que la fonction fait ce qu'on attend d'elle
  - ex : `node:assert` (natif à node mais récent) (`assert.isEqual(result, 42)`)
  - ex : `chai` (+ plugin)
  - ex : `jest`
  - ex : `vitest`


### Comparaison test runners

(Culture G)

| Outil           | Type                | Points forts                                                                                                                                      | Limites / Inconvénients                                                                   | Communautés & Plugins              |
| --------------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- | ---------------------------------- |
| **`node:test`** | Natif Node.js       | - Inclus par défaut dès Node.js 18+ <br> - Minimal, rapide, sans dépendance                                                                       | - Peu de fonctionnalités avancées (ex: mocking)<br> - Peu intégré aux outils modernes     | Faible (car très bas niveau)       |
| **`mocha`**     | Lib JS indépendante | - Très flexible<br> - Très stable & mature <br> - Compatible avec `chai` / `sinon`                                                                | - Nécessite de plugger assertions / mocks à la main <br> - Configuration parfois verbeuse | Large (via plugins & middlewares)  |
| **`jest`**      | Framework complet   | - Batteries incluses : runner, assertions, mocks, coverage <br> - API simple et expressive <br> - Super bien intégré React / TS                   | - Plus lourd <br> - Moins rapide que `vitest` pour tests unitaires purs                   | Très large, support officiel React |
| **`vitest`**    | Framework complet   | - Ultra rapide grâce à Vite (ESM, hot reload) <br> - Très similaire à Jest (API / compatibilité) <br> - Excellent avec ESM, TS, Vue, Svelte, etc. | - Plus jeune, donc parfois moins mature en mocking avancé                                 | Communauté en croissance rapide    |

### Comparaison bibliothèque d'assertion

(Culture G)

| Outil             | Type                 | Points forts                                                                                                 | Limites / Inconvénients                                | API                         |
| ----------------- | -------------------- | ------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------ | --------------------------- |
| **`node:assert`** | Natif Node.js        | - Inclus dans Node.js <br> - Aucune dépendance <br> - Simple & fiable                                        | - API très basique <br> - Pas d'API fluide (chainable) | `assert.strictEqual()` etc. |
| **`chai`**        | Lib JS indépendante  | - Syntaxe fluide et lisible (`expect`, `should`) <br> - Extensible avec plugins (e.g. HTTP, promises, spies) | - À associer à un runner et spy/mocking externe        | `expect(foo).to.equal(42)`  |
| **`jest`**        | Intégré au framework | - API claire et riche <br> - Inclus avec Jest runner <br> - Fonctionne bien avec TS                          | - Couplé au runner Jest                                | `expect(foo).toBe(42)`      |
| **`vitest`**      | Intégré au framework | - API identique à Jest <br> - Support TS natif <br> - Assertions modernes                                    | - Moins de plugins que Chai                            | `expect(foo).toBe(42)`      |

## Tests d'intégration

= component tests
= specifications tests (`spec`)

Ils testent une partie du système : l'API.

Mais comment ? Car actuellement pour tester manuellement, on doit :
- créer une base de données (oquiz) de développement
- gérer les migrations
- lancer un serveur pour l'app (port 3000)
- tester à l'aide de postman

Donc :
- quelle BDD doit utiliser nos tests ? la même ? donc on est forcé d'installer Postgres en local pour run les tests ? 
  - si le test en question écris dans la BDD, on fou en l'air le seeding
  - si le test en question lit la BDD de dev, on peut plus modifier la BDD de dev sans casser le test ?
- quel serveur HTTP utilise nos tests ? le port 3000 ? 
  - sous entendu je dois lancer `npm run dev --prefix api` dans un autre terminal AVANT de lancer les tests ?


SOLUTION : on va rajouter un setup, qui doit :
- créer une BDD dédiée au test (que l'on supprime potentiellement une fois les tests lancés)
  - à l'aide d'un conteneur 
- lancer un serveur HTTP de test
  - dans le setup, on lance `app.listen`

Ce travail de setup est à faire une fois, généralement en entreprise, c'est déjà là !
- => c'est un travail d'architecture (pas si évident) 


## Spread operator

```js
const person = {
  pseudo: "Bob",
  age: 42
};

const address = {
  numero: 1,
  road: "rue des crustacés",
  city: "Bikini Bottom"
};


const total = {
  ...person,
  ...address
};

console.log(total); // { pseudo, age, numero, road, city }
```


## Ternary operator

```js
console.log(Math.random() > 0.5 ? "grand" : "petit");
//             ^ si vrai, alors l'ensemble de l'expression vaudra "grand"
//             ^ si faux, alors l'ensemble de l'expression vaudra "petit"
```


## Démarche : ajout de nouveaux endpoints 

Base de données :
- 1) Modifier le schéma prisma
- 2) Migrations : `npm run db:migrate:dev --prefix api`
- 3) Seeding à modifier : `seeding.ts` 
- 4) Reset la BDD : `npm run db:reset` pour ajouter le nouveau seeding

Router :
- 1) Rajouter un router
- 2) Brancher au router d'index

Controller : 
- 1) Créer le controleur et une première méthode
- 2) Faire un controleur qui ne renvoie rien (ou un simple objet pour tester)
- 3) Tester avec Postman
- 4) Commit

Implémenter le controlleur : 
- 1) en testant régulièrement sur postman
- 2) ajouter un test "spec" a minima (celui qui vous semble le plus pertinent)
