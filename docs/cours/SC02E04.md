## Zod validation

- `parse` : valide le body, et s'il ne respecte pas le format, il lève une erreur (`throw`) ==> on peut catch l'erreur
- `parseAsync` : idem, mais en version `async` (=> donc il faut `await`)

- `safeParse` : valide le body, et s'il ne respecte pas le format, il renvoie une erreur (`return`) ==> on peut récupérer l'erreur
- `safeParseAsync` : idem mais en version `async` (=> donc il faut `await`)

En savoir plus sur l'event loop : 
- [What the heck is the event loop anyway](https://www.youtube.com/watch?v=8aGhZQkoFbQ&ab_channel=JSConf)
- [JavaScript Visualized: Event Loop, Web APIs, (Micro)task Queue](https://www.lydiahallie.com/blog/event-loop)

## Déroulée de la journée 

Corriger les routes des `/levels` progressivement. Avec pour objectif d'atteindre une architecture agréable en terme de gestion des erreurs : 
- Les erreurs `Zod` doivent remonter au `globalErrorHandler` et renvoyer une `422`
- Les erreurs maitrisées (`HttpErrors`) doivent remonter au `globalErrorHandler` et le status code approprié
- Les autres erreurs (non maitrisées) doivent remonter au `globalErrorHandler` pour être loggé (erreur) et renvoyer une `500`.

En bref, on passe de : 
- `return res.status(409).json({ error: "..." })` à `throw new ConflictError("...")`
- `const { error, data } = zodSchema.safeParse(req.body)` à `const data = await zodSchema.parseAsync(req.body)`

## Automatisation

**CI/CD** = processus réguliers d'intégration et de déploiement

- CI = **Continuous Integration** = l'intégration continue = l'amélioration continue du code
  - PR = Pull Request = intègre du code d'une branche dans une autre
  - on souhaite s'assurer qu'il n'y a pas de regression, pas de baisse de qualité de code
    - éviter les regressions : tests automatisés
    - garantir la qualité du code : typescript, eslint

- CD = **Continuous Deployement** = déploiement continue = mise en production automatisés du code
  - Branche `prod` = à chaque fois qu'elle est mise à jour, re-déploie automatiquement le code
  - outils possibles : 
    - build automatiquement l'image du code et push vers dockerhub
    - webhook sur le serveur de production (système qui écoute des changements sur github et pull automatique)
  - exemple de déploiement continue
    - GitHub Pages : un push sur une branche, la page est redéployé
    - Vercell, Heroku, Railway (PaaS) proposent généralement du déploiement continue

## Husky

Husky est un util qui permet de **gérer facilement des hooks Git** (scripts exécutés automatiquement à certaines étapes du cycle Git, comme `pre-commit`, `pre-push`, etc.) directement depuis ton projet.

En clair, il sert à **exécuter automatiquement des actions avant ou après certaines commandes Git** pour renforcer la qualité du code et éviter d’envoyer du mauvais code sur le dépôt.

Par exemple : 

- **Exécuter des tests avant un commit**
  - Empêche de valider du code qui casse l'application.
- **Lancer un linter (ex: ESLint) sur les fichiers modifiés**
  - Garantit un code propre et homogène.
- **Formatter le code avec Prettier avant de pousser**
  - Évite les conflits liés au style.
- **Vérifier les messages de commit**
  - Applique un format de commit (ex: Conventional Commits).
- **Bloquer un commit/push si les tests échouent**
  - Assure la qualité du dépôt.


## GitHub Actions

**GitHub Actions** est un service d'intégration et de déploiement continus (CI/CD) intégré directement à GitHub. Il permet d'automatiser des workflows comme :

* **Tester automatiquement le code** à chaque *push* ou *pull request*
* **Construire (build)** des applications
* **Déployer** sur des serveurs, cloud, ou autres services
* **Lancer des scripts de maintenance ou d'audit**

### Fonctionnement

Un **workflow GitHub Actions** est défini par un fichier YAML dans le dossier `.github/workflows` de votre dépôt.

Il est composé de :

* **Événements déclencheurs** : push, pull\_request, cron (programmation régulière), tag, release...
* **Jobs** : une suite d'étapes qui s'exécutent sur un runner (machine virtuelle mise à disposition par GitHub ou auto-hébergée).
* **Steps** : chaque étape peut exécuter un script ou une **action** (des fonctionnalités réutilisables créées par la communauté ou GitHub).

### Exemple minimal

```yaml
name: Run Tests

on: [push]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Install dependencies
        run: npm install
      - name: Run tests
        run: npm test
```

### Avantages

* **Intégré à GitHub** : pas besoin d'outil externe.
* **Facile à configurer**.
* **Marketplace d’actions** déjà prêtes.
* **Gratuit dans certaines limites** pour les dépôts publics et plans gratuits.

### Cas d’usage courants

* Lancer des tests unitaires ou E2E
* Construire des artefacts (ex : Docker image)
* Publier sur npm, PyPI, etc.
* Déployer sur AWS, GCP, Azure, Vercel, Netlify, etc.
* Vérifier le formatage ou la qualité du code (linting, sécurité...)
